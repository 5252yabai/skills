# Session Ingest

AI 대화 세션을 원본 소스로 삼아 위키를 생성·업데이트하는 워크플로우.
`wiki-ingest`와 동일한 위키 처리를 하되, 원본이 파일이 아니라 현재 대화 세션이다.

## 위키 경로

위키 Vault의 절대 경로: `/Users/leejaeseok/texts_wiki`

모든 파일 읽기·쓰기는 이 경로 기준으로 수행한다. 현재 작업 디렉토리와 무관하다.

## 위키 규칙 (CLAUDE.md 요약)

- `1.Raw/`는 저장 후 **절대 수정하지 않는다**
- AI 세션 원본은 `1.Raw/{YYYY-MM}/ai_session/`에만 저장한다
- `2.Wiki/`만 관리한다 (entities/, themes/, analyses/)
- `[[문서명]]` 형식 wikilink 사용
- Category/Topic/Tag 분류 체계 **금지** — wikilink만으로 조직
- 한국어로 작성, H1(`#`) 금지 — `##`(H2)부터 시작
- 각 페이지 하단에 `## 관련 원본` 섹션 유지

## 핵심 원칙

- **원본 무가공 저장**: 대화 내용을 어떠한 편집·요약·재구성 없이 그대로 `1.Raw/`에 저장한다
- **1.Raw/ 읽기 전용**: 저장 후 수정하지 않는다

## 절차

### 1단계: 세션 원본 저장

대화 세션의 핵심 내용을 **가공 없이** `1.Raw/`에 저장한다.

**파일 경로**: `1.Raw/{YYYY-MM}/ai_session/세션제목.md`

- 연-월 폴더는 오늘 날짜 기준
- 연-월 폴더 아래 `ai_session/` 폴더를 생성·사용
- 세션 제목은 대화의 핵심 주제를 간결하게 (예: `이슈-정리-세션`)

**파일 구조**:

```markdown
---
created: { YYYY-MM-DD }
source: ai-session
session_id:
  {
    세션 ID — ~/.claude/projects/ 하위 현재 프로젝트 디렉터리에서 가장 최근 .jsonl 파일의 basename,
  }
ai_model: { 현재 세션의 AI model. 모르면 unknown }
ai_effort: { 현재 세션의 effort/thinking level. 모르면 unknown }
---

# {세션 제목}

## 핵심 의사결정

{사용자가 세션 중 내린 주요 결정을 불릿 리스트로 정리}
{각 항목은 "무엇을 결정했는가" + "왜/어떤 맥락에서"로 구성}
{예: "- useMemo 제네릭 제거 선택 — Provider props에서 타입이 강제되므로 중복"}
{예: "- 1번 방안(콜백 주입) 채택 — feature 컨벤션 위반으로 패키지화 불가"}

## 세션 내용

{대화에서 다룬 내용을 원문 그대로 기록}
{사용자 발언, AI 응답, 의사결정, 작업 결과 등을 시간 순서대로}
{요약하지 않는다. 대화의 맥락이 보존되어야 한다}
```

**Frontmatter 규칙**:

- 허용 필드: `created`, `source`, `session_id`, `ai_model`, `ai_effort`만 사용한다
- `source`는 원천 구분용이다. 세션 ingest는 항상 `source: ai-session`으로 기록한다 (다른 원천이 추가되면 그에 맞는 source 값 사용)
- `session_id`는 현재 AI 세션의 고유 식별자다. `~/.claude/projects/` 하위 현재 프로젝트 디렉터리에서 가장 최근 수정된 `.jsonl` 파일의 basename(확장자 제외)이 세션 ID다. 예: `4c63bcb6-4ead-42cc-b51f-774aeea3f7f5`
- `ai_model`은 현재 세션을 수행한 AI model이다. 확인 불가하면 `unknown`으로 기록한다
- `ai_effort`는 현재 세션의 effort/thinking level이다. 확인 불가하면 `unknown`으로 기록한다
- `topics`, `objects` 같은 분류 필드는 **사용하지 않는다** — 위키 분류는 `2.Wiki/`의 wikilink로만 한다

**원본 저장 기준**:

- 사용자의 요청과 의사결정은 반드시 포함
- AI가 조사·분석한 결과 데이터 포함
- 코드나 설정 변경 내역 포함
- 단순 확인 응답("Y", "ㄱㄱ")은 의사결정 맥락으로 포함
- MCP 도구 호출의 raw 응답은 제외하되, 핵심 결과는 포함

### 2단계: 위키 처리

저장된 원본을 기반으로 아래 절차를 수행한다.

1. **핵심 정보 추출** — 세션에서 다룬 주요 주제, 언급된 개체, 의사결정 식별
2. **Entity 페이지 처리** — 언급된 개체의 `2.Wiki/entities/` 페이지 업데이트 또는 [entity 템플릿](assets/entity.md) 기반 생성
3. **Theme 페이지 처리** — 관련 주제의 `2.Wiki/themes/` 페이지 업데이트 또는 [theme 템플릿](assets/theme.md) 기반 생성
4. **Analysis 페이지** — 보존할 가치가 있는 교차 분석이 있으면 [analysis 템플릿](assets/analysis.md) 기반 생성
5. **연결 발견** — 기존 원본·위키와의 관련성 탐색, wikilink 기록
6. **인덱스·로그 업데이트** — `2.Wiki/index.md` 등록 (가나다순), `2.Wiki/log/YYYY-MM.md` 기록 (최신이 위). 로그 형식은 `- 작업유형: 설명 ([[관련/문서]], [[1.Raw/원본]])`. 작업유형 대괄호 금지. 문서 경로는 backtick 대신 wikilink 사용. wikilink 문법 충돌 시 클릭 가능한 Markdown 링크 사용.

위 2~4단계에서 기존 위키 내용과 충돌이 보이면 아래 [충돌·정확성 처리](#충돌정확성-처리)를 따른다.

### 충돌·정확성 처리

위키에 내용을 생성·덧붙일 때 기존 내용과 충돌이 발견되면 두 경우로 구분한다.

- **관점·시간 변화로 인한 모순** (예: 과거엔 A를 선호 → 지금은 B): 정상적 사유 변화다. 양쪽을 **그대로 기록**하고 변화 맥락을 명시한다.
- **사실관계 충돌 + 정확성 불확실** (예: 세션·원본마다 날짜·수치·주체가 다름, 어느 쪽이 맞는지 판단 불가): 추측으로 덮어쓰지 말고 **사용자에게 질문**한 뒤 답변을 반영한다.

질문할 때는 어느 원본의 어떤 값이 충돌하는지 구체적으로 제시한다. 명백한 오타 등 확실히 판단 가능한 경우는 질문 없이 처리한다.

### 3단계: 결과 보고

처리 결과를 사용자에게 보고한다:

- 저장된 원본 파일 경로
- 생성/업데이트된 위키 페이지 목록
- 새로 발견된 연결

## 주의사항

- 원본 저장이 **반드시 먼저** 완료된 뒤 위키 처리를 진행한다
- 세션 원본은 항상 `1.Raw/{YYYY-MM}/ai_session/`에 저장한다
- 원본에는 대화 내용을 가공하지 않는다
- 한 세션에서 여러 주제를 다뤘으면 하나의 원본 파일에 모두 포함한다
- 사용자가 세션의 특정 부분만 지정하면 해당 부분만 처리한다
